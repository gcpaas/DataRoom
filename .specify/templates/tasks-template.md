# 任务清单：[功能名称]

**输入**: 来自 `/specs/[###-feature-name]/` 的设计文档

**前置条件**: plan.md（必需）、spec.md（用户故事必需）、research.md、data-model.md、contracts/

**验证要求**: 下列示例包含验证任务。凡是被宪章或功能影响面触发的验证任务都属于强制项。
前端改动需要 `npm run type-check`；前端样式敏感改动还需要 `npm run lint`；后端逻辑改
动需要相关 Maven 测试；Java 改动触及 `catch` 块时需要 `CatchBlockLoggingTest`。

**组织方式**: 任务按用户故事分组，确保每个故事都能独立实现与独立测试。

## 格式：`[ID] [P?] [Story] 任务描述`

- **[P]**: 可并行执行（不同文件、无依赖）
- **[Story]**: 所属用户故事（如 US1、US2、US3）
- 描述中必须写明准确文件路径

## 路径约定

- **单体项目**: 仓库根目录下使用 `src/`、`tests/`
- **Web 应用**: 使用 `backend/src/`、`frontend/src/`
- **移动端**: 使用 `api/src/`、`ios/src/` 或 `android/src/`
- 下方示例默认采用单体项目路径，生成任务时应按 plan.md 中的真实结构调整

## 依赖与执行顺序

### 阶段依赖

- **准备阶段（Phase 1）**: 无依赖，可立即开始
- **基础阶段（Phase 2）**: 依赖准备阶段完成，并阻塞所有用户故事
- **用户故事阶段（Phase 3+）**: 全部依赖基础阶段完成
  - 如团队有能力，可在此后并行推进多个用户故事
  - 也可按优先级顺序执行（P1 → P2 → P3）
- **收尾阶段（最终阶段）**: 依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**: 基础阶段完成后即可开始，不依赖其他用户故事
- **用户故事 2（P2）**: 基础阶段完成后即可开始，可能与 US1 集成，但仍应可独立测试
- **用户故事 3（P3）**: 基础阶段完成后即可开始，可能与 US1/US2 集成，但仍应可独立测试

### 每个用户故事内部顺序

- 只要涉及行为或后端逻辑变更，测试 MUST 先写且先失败
- 先模型，再服务
- 先服务，再接口/功能入口
- 用户故事签收前必须完成契约同步更新
- 用户故事签收前必须完成要求的验证
- 先完成核心实现，再做集成收尾
- 当前故事完成后，再进入下一个优先级故事

### 并行机会

- 所有标记为 [P] 的准备任务都可并行
- 基础阶段内所有标记为 [P] 的任务都可并行
- 基础阶段完成后，多个用户故事可并行推进（若团队容量允许）
- 同一用户故事下标记为 [P] 的测试可并行
- 同一用户故事下标记为 [P] 的模型任务可并行
- 不同成员可并行处理不同用户故事

## 实施策略

### 先做 MVP（仅用户故事 1）

1. 完成 Phase 1：准备阶段
2. 完成 Phase 2：基础阶段（关键阻塞项）
3. 完成 Phase 3：用户故事 1
4. **停止并验证**：独立测试用户故事 1
5. 若已满足条件，则部署或演示

### 增量交付

1. 完成准备阶段 + 基础阶段，打通底座
2. 交付用户故事 1 → 独立测试 → 部署/演示（MVP）
3. 交付用户故事 2 → 独立测试 → 部署/演示
4. 交付用户故事 3 → 独立测试 → 部署/演示
5. 每个故事都在不破坏已有价值的前提下逐步增量

### 并行团队策略

在多人协作场景下：

1. 团队共同完成准备阶段和基础阶段
2. 基础阶段完成后：
   - 开发者 A：处理用户故事 1
   - 开发者 B：处理用户故事 2
   - 开发者 C：处理用户故事 3
3. 各用户故事独立完成并逐步集成

## 说明

- `[P]` 任务表示不同文件、无依赖，可并行处理
- `[Story]` 标签用于将任务与具体用户故事建立追踪关系
- 每个用户故事都应可独立完成并独立测试
- 实现前要先确认必需测试已处于失败状态
- 每完成一个任务或一个合理任务组后提交一次
- 在任何检查点都可以停下并独立验证当前故事
- 避免：任务描述模糊、多个任务改同一文件冲突、破坏独立性的跨故事耦合
