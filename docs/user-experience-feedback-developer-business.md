# DataRoom-Plus 使用反馈与优化建议

## 1. 背景

本文从两类典型使用者视角梳理 DataRoom-Plus 在实际使用过程中可能遇到的问题，并给出优化建议：

- 开发人员：具备前后端、SQL、接口、数据源、部署或二次开发能力，关注效率、可调试性、可扩展性和稳定性。
- 业务人员：不会开发或只具备少量开发经验，关注上手成本、配置可理解性、操作容错、模板复用和交付效率。

反馈重点站在用户体验和实际使用流程角度，不讨论底层技术实现细节。

## 2. 开发人员视角：10 条使用反馈与优化建议

| 序号 | 使用反馈 / 可能遇到的问题 | 影响 | 优化建议 |
|------|----------------------------|------|----------|
| 1 | 数据源、数据集、页面组件之间的依赖关系不够直观。开发人员排查某个组件数据异常时，需要在多个页面之间来回跳转。 | 问题定位成本高，尤其在复杂大屏中难以快速判断是数据源、SQL、参数、字段映射还是组件渲染问题。 | 增加“依赖关系视图”或“组件数据链路面板”，展示页面 -> 组件 -> 数据集 -> 数据源 -> 参数 -> 字段映射的完整链路，并支持一键跳转。 |
| 2 | 数据集调试能力不足。SQL、HTTP、JSON 数据集虽然能执行，但缺少更完整的调试上下文。 | 开发人员难以快速复现线上组件数据异常，也难以判断参数替换、响应解析、字段映射是否正确。 | 为数据集增加调试控制台，展示入参、实际执行 SQL/HTTP 请求、响应原文、JSONPath 结果、字段映射结果、耗时和错误堆栈。 |
| 3 | 组件配置结构分散在 `install.ts`、`index.vue`、`panel/index.vue` 中，新增或修改组件时容易遗漏同步。 | 组件扩展门槛偏高，容易出现默认值、渲染逻辑、配置面板不一致的问题。 | 增加组件开发脚手架或组件配置校验工具，自动生成标准目录、默认配置、面板占位和类型定义，并在构建时检查配置字段一致性。 |
| 4 | 多态 JSON 配置依赖注解维护，新增页面、数据源、数据集子类型时容易漏配。 | 反序列化失败通常在运行期暴露，错误信息对使用者不够友好。 | 增加启动期配置自检，扫描已注册子类型并输出明确错误；接口返回中说明缺失的类型标识、目标类和建议修复位置。 |
| 5 | 前后端错误提示偏通用，很多异常只表现为“请求失败”或后端异常信息。 | 开发人员需要打开日志、浏览器控制台和接口响应反复确认问题。 | 建立统一错误码和错误说明，前端展示用户可理解的信息，同时提供“查看技术详情”入口，包含 traceId、接口地址、请求参数摘要和后端错误码。 |
| 6 | 页面发布、预览、设计态之间的版本差异不够清晰。 | 开发人员在协助业务排查时，不容易判断当前查看的是草稿、预览版本还是已发布版本。 | 增加版本状态标识、发布时间、发布人、草稿更新时间，并提供设计态与发布态配置 diff 查看能力。 |
| 7 | 权限角色较少，开发人员、业务编辑、查看者之间的操作边界不够细。 | 团队协作时容易出现权限过大或权限不足，例如业务人员误改数据源、查看者误进入设计页面。 | 将权限拆分为页面查看、页面编辑、发布、数据源管理、数据集管理、资源管理、用户管理等细粒度权限，并在界面中隐藏无权限入口。 |
| 8 | 资源文件、页面配置和数据集之间缺少变更影响提示。 | 删除图片、接口、数据集或字段时，可能导致已有大屏静默失效。 | 在删除或修改关键资源前执行影响分析，提示受影响页面、组件和字段；对高风险操作提供确认清单。 |
| 9 | 缺少面向部署和运维的健康检查视图。 | 开发人员需要从日志、数据库、接口文档等多个入口判断系统是否正常。 | 增加系统诊断页面，展示后端版本、前端版本、数据库连接、资源目录、数据源连通性、定时刷新状态和最近错误。 |
| 10 | 二次开发文档偏分散，新开发者需要从代码约定中推断很多规则。 | 接入成本高，组件开发、数据集扩展、数据源扩展容易踩坑。 | 增加“开发者中心”文档，集中说明组件开发流程、数据集扩展流程、数据源扩展流程、接口规范、类型注册规则、常见错误和示例代码。 |

## 3. 业务人员视角：10 条使用反馈与优化建议

| 序号 | 使用反馈 / 可能遇到的问题 | 影响 | 优化建议 |
|------|----------------------------|------|----------|
| 1 | 创建大屏时缺少明确的新手引导。业务人员进入设计器后，面对画布、组件库、配置面板、数据集配置容易不知道第一步做什么。 | 上手门槛高，容易依赖开发人员协助完成基础页面搭建。 | 增加首次使用引导，按“选择模板 -> 添加组件 -> 绑定数据 -> 预览 -> 发布”的流程提供分步提示和示例页面。 |
| 2 | 数据集、数据源、字段映射等概念偏技术化。 | 业务人员不理解“数据集”和“字段映射”的关系，配置图表时容易出错。 | 使用更贴近业务的文案，例如“数据来源”“展示字段”“筛选条件”；在配置面板中增加字段含义说明和示例值预览。 |
| 3 | 空白画布缺少可直接复用的模板。 | 业务人员从 0 搭建大屏耗时长，页面布局和视觉效果不稳定。 | 提供行业模板、常用布局模板和组件组合模板，例如经营驾驶舱、设备监控、销售看板、告警中心等，并支持一键替换数据。 |
| 4 | 组件配置项较多，且技术配置和常用配置混在一起。 | 业务人员很难判断哪些配置必须改、哪些可以保持默认，容易误改高级参数。 | 将配置面板分为“常用”“数据”“样式”“高级”几类，默认只展开常用配置，高级配置增加说明和恢复默认能力。 |
| 5 | 数据绑定后的效果反馈不够即时。 | 业务人员配置字段后，需要反复预览或刷新才能判断图表是否正确。 | 在组件配置面板中增加实时数据预览、字段匹配状态和异常提示，例如“字段不存在”“返回为空”“数据类型不匹配”。 |
| 6 | 拖拽、缩放、对齐操作对非设计人员不够友好。 | 页面元素容易不整齐，多个组件之间间距不一致，影响最终展示效果。 | 增加智能对齐线、等距分布、一键吸附、批量对齐、锁定比例、复制样式和布局整理功能。 |
| 7 | 保存、预览、发布之间的区别不够明显。 | 业务人员可能以为保存后观众就能看到，也可能误发布未完成页面。 | 在界面上明确区分“草稿已保存”“预览仅自己可见”“发布后对外可见”，发布前展示确认弹窗和变更摘要。 |
| 8 | 出错时提示不够面向业务。 | 业务人员看到接口错误、SQL 错误、JSONPath 错误时不知道如何处理，只能求助开发。 | 错误提示使用业务语言描述，例如“当前图表没有拿到数据，请检查数据来源是否可用”；同时提供“复制错误信息给开发人员”的按钮。 |
| 9 | 页面资产管理和复用能力不足。 | 业务人员难以复用已有图片、背景、图标、组件样式和页面片段。 | 增加资源库和收藏能力，支持常用背景、图标、组件组合、页面片段、主题样式的复用和团队共享。 |
| 10 | 协作和审批流程不明显。 | 多个业务人员共同编辑时容易覆盖修改，发布前也缺少确认机制。 | 增加编辑锁、协作者提示、修改记录、版本回退和发布审批流程，适合多人协作和正式交付场景。 |

## 4. 综合优化方向

1. 降低业务人员上手门槛：优先建设模板、引导、业务化文案、实时预览和错误友好提示。
2. 提升开发人员排查效率：优先建设数据链路、调试控制台、错误码、版本 diff 和系统诊断。
3. 强化协作安全性：完善权限、影响分析、版本管理、发布确认和多人编辑机制。
4. 提升扩展一致性：为组件、数据集、数据源扩展提供脚手架、校验工具和集中式开发文档。
5. 将“能配置”升级为“可放心交付”：围绕模板复用、配置校验、发布流程和回滚能力完善产品闭环。

## 5. 建议优先级

| 优先级 | 建议事项 | 原因 |
|--------|----------|------|
| P0 | 错误提示优化、数据集调试、保存/预览/发布状态区分 | 直接影响日常使用和问题排查，是最容易产生阻塞的环节。 |
| P1 | 新手引导、模板库、字段映射预览、组件配置分层 | 能显著降低业务人员使用门槛，提高页面搭建效率。 |
| P2 | 版本 diff、影响分析、权限细分、资源复用 | 适合在多人协作和正式交付场景中提升稳定性。 |
| P3 | 组件脚手架、开发者中心、系统诊断、发布审批 | 面向中长期扩展、运维和团队化建设。 |
