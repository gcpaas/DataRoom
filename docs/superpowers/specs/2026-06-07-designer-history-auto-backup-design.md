# 设计器历史自动备份设计

## 背景

DataRoom 当前有两个编辑器：

- `PageDesigner`：网格化页面设计器
- `VisualScreenDesigner`：像素级大屏设计器

两个编辑器当前都依赖设计态 `pageConfig` 作为唯一持久化数据。用户如果遇到断电、浏览器崩溃、标签页误关闭，且本次编辑还没有手动点击保存，就会丢失当前设计内容。

后端已经存在 `dr_page_stage` 中转态表，以及历史列表和历史回退相关接口，但现有能力有两个明显缺口：

- `updatePageConfig` 仅覆盖设计态，不会自动为设计器生成历史备份
- 现有 `/stage/rollback/{id}` 通过切换 `pageStatus` 完成回退，不是“复制当前态为历史，再复制目标历史覆盖设计态”的快照式回退

本次需求要补齐设计器侧的历史自动备份能力，并把历史查看与回滚能力接入两个设计器顶部工具栏。

## 目标

- 为 `PageDesigner` 和 `VisualScreenDesigner` 增加统一的历史自动备份能力
- 手动点击保存时，始终额外新增一条历史记录，即使内容与上一次保存完全相同
- 前端每分钟检查一次当前配置；如果与最近一次自动备份配置不同，则新增一条自动备份历史
- 在两个设计器顶部 `预览` 按钮左侧新增 `历史` 按钮
- 点击 `历史` 后以弹窗方式分页展示历史记录，按时间倒序排列
- 用户点击某条历史后的 `回滚` 按钮时，先确认，再执行“回滚前备份当前设计态 + 目标历史覆盖设计态”
- 回滚成功后页面自动刷新，保证编辑器重新加载最新设计态

## 非目标

- 本期不限制历史数量，不增加自动清理策略
- 本期不做“进入设计器时自动检测是否存在较新的自动备份并弹提示”
- 本期不做历史 diff 预览
- 本期不做本地浏览器缓存备份
- 本期不区分设计器历史与发布历史，历史弹窗直接展示所有 `history` 状态记录
- 本期不修改发布、下线现有写历史的行为

## 已确认的产品规则

- 手动保存时，每次都必须新增一条历史记录
- 自动备份不限数量，先全部保留
- 历史弹窗展示当前页面下所有 `history` 记录，包括发布、下线产生的历史
- 如果当前编辑器存在未保存改动，则禁止直接回滚，必须先保存
- 历史列表采用分页表格，默认每页 20 条

## 现状问题

### 现有历史表可复用

后端已经有 `PageStageEntity`，并通过 `pageStatus` 区分 `design`、`history`、`published`、`preview`、`snapshot`。这意味着本次不需要新增历史表，直接复用 `dr_page_stage` 即可。

### 现有回滚语义不满足需求

当前 `PageController.stageRollback` 的行为是：

1. 将当前 `design` 记录改成 `history`
2. 将目标 `history` 记录改成 `design`

这种实现有几个问题：

- 目标历史记录本身会被“转正”为设计态，不再保留为原始历史快照
- 回滚后的“当前设计态”没有明确生成一条新的历史副本
- 它依赖记录身份切换，而不是依赖配置内容复制，回滚语义不稳定

本次需求需要的是真正的快照式回滚：

1. 把当前设计态复制新增为一条新的 `history`
2. 把目标历史的 `pageConfig` 复制覆盖到当前 `design`
3. 保留目标历史原记录不变

## 总体方案

采用“后端历史快照为唯一真源，前端定时触发”的方案。

职责划分如下：

- 后端负责：
  - 创建历史记录
  - 查询历史列表
  - 执行快照式回滚
- 前端负责：
  - 生成当前编辑器配置快照
  - 判断当前是否存在未保存改动
  - 每分钟触发一次自动备份检查
  - 展示历史弹窗与回滚交互

这样可以确保两个设计器使用同一套历史规则，后续也方便接入操作日志、清理策略和历史筛选能力。

## 后端设计

### 数据模型

继续复用 `dr_page_stage`，本期不改表结构，不新增字段。

历史来源通过 `remark` 表达：

- 手动保存备份：`手动保存自动备份`
- 自动备份：`自动备份`
- 回滚前备份：`回滚前自动备份`
- 发布、下线等已有历史继续保留原始备注

这样做的原因是：

- 当前需求不要求按来源筛选
- 弹窗要展示所有 `history`
- 首版不值得为了来源分类单独改表和补兼容脚本

### 接口设计

#### 1. 新增历史备份接口

`POST /dataRoom/page/history/backup`

请求体：

```json
{
  "pageCode": "page_xxx",
  "pageType": "page",
  "remark": "自动备份",
  "pageConfig": {}
}
```

处理规则：

1. 校验页面编码和页面类型
2. 基于请求体创建一条新的 `PageStageEntity`
3. `pageStatus` 固定写为 `history`
4. `remark` 使用前端传入值
5. `pageConfig` 使用请求体快照内容
6. 保存后返回新增记录主键

说明：

- 此接口既服务手动保存后的自动备份，也服务一分钟自动备份
- 不在接口内做内容去重，是否需要备份由前端调用前判断
- 即使请求体内容与已有历史完全一致，只要调了接口，也必须新增历史

#### 2. 历史列表接口

继续复用现有：

`GET /dataRoom/page/stage/list`

前端固定传：

- `code=当前页面编码`
- `pageStatus=history`
- `current=页码`
- `size=20`

服务端需要保证结果按 `createDate desc` 返回。如果当前接口没有显式排序，需要在本次实现中补上排序，避免列表顺序不稳定。

#### 3. 快照式回滚接口

新增：

`POST /dataRoom/page/history/rollback/{id}`

处理步骤固定为：

1. 根据 `id` 查询目标历史记录
2. 校验目标记录存在，且 `pageStatus=history`
3. 根据 `pageCode + design` 查询当前设计态记录
4. 将当前设计态复制新增为一条新的 `history`，备注固定为 `回滚前自动备份`
5. 用目标历史的 `pageConfig` 覆盖当前设计态记录的 `pageConfig`
6. 更新当前设计态记录的 `updateDate`
7. 返回成功

关键要求：

- 目标历史记录自身不改状态
- 当前设计态记录自身不改状态
- 回滚是“复制配置内容”，不是“交换记录身份”

#### 4. 旧回滚接口处理

现有 `/dataRoom/page/stage/rollback/{id}` 不应再作为设计器历史回滚入口使用。

建议：

- 保留旧接口，避免影响未知调用方
- 设计器前端新功能全部改走 `/dataRoom/page/history/rollback/{id}`
- 在代码注释中标明旧接口为旧语义，不再用于设计器历史回滚

### 后端服务边界

建议把历史备份和回滚逻辑收敛到 `PageStageService` 或新建一个历史应用服务，避免控制器直接堆叠复制逻辑。

最小边界如下：

- `backupHistory(pageCode, pageType, pageConfig, remark)`
- `rollbackDesignByHistoryId(historyId)`

这样有两个好处：

- 保存和回滚逻辑能被多个控制器入口复用
- 测试可以直接覆盖服务层，而不是全部走控制器分支

### 操作日志

本期新增的备份和回滚接口应继续接入操作日志。

建议：

- 历史备份：`businessType = "page_stage"`，`businessName = "页面历史"`
- 历史回滚：`businessType = "page_stage"`，`businessName = "页面历史"`

回滚继续记录为修改类操作。

## 前端设计

### 共享配置快照生成

两个设计器都已经有生成保存 payload 的逻辑：

- `PageDesigner.vue`：`chartList + pageBasicConfig + globalVariable`
- `VisualScreenDesigner.vue`：`chartList + basicConfig + globalVariable`

本期需要再抽一层共享规则：

- 能拿到“当前设计器完整 pageConfig”
- 能将其稳定序列化
- 能计算摘要值

前端判断差异不直接使用对象引用或普通 `JSON.stringify`。应使用稳定序列化后再计算 `md5`。

原因：

- 对象键顺序不稳定时，普通序列化可能误判变化
- 摘要值可同时服务未保存判断和自动备份判断

### 前端本地状态

两个设计器各自维护以下状态：

- `lastSavedHash`
  - 最近一次成功保存设计态后的配置摘要
- `lastAutoBackupHash`
  - 最近一次成功自动备份后的配置摘要
- `historyDialogVisible`
  - 历史弹窗开关
- `historyList`
  - 当前页历史数据
- `historyLoading`
  - 历史弹窗加载状态
- `historyPagination`
  - 页码、每页条数、总数

### 未保存改动判断

规则固定为：

- `currentHash !== lastSavedHash`，说明当前有未保存改动
- `currentHash !== lastAutoBackupHash`，说明距离最近一次自动备份已经发生变化

这两种判断必须分开，因为：

- 用户可能已经手动保存，但还没触发自动备份
- 用户可能已经自动备份，但还没点手动保存

本次回滚拦截只看“是否有未保存改动”，因此只比较 `currentHash` 和 `lastSavedHash`。

### 手动保存流程

点击 `保存` 后的流程改为：

1. 生成当前 `pageConfig`
2. 调用 `pageApi.updatePageConfig`
3. 保存成功后，再调用 `pageApi.historyBackup`
4. 历史备注固定为 `手动保存自动备份`
5. 两个请求都成功后，更新：
   - `lastSavedHash = currentHash`
   - `lastAutoBackupHash = currentHash`
6. 提示 `保存成功`

之所以手动保存成功后同时刷新 `lastAutoBackupHash`，是因为这次“手动保存自动备份”已经产生了一条最新历史，没有必要让下一分钟定时器再对相同内容重复新增一条 `自动备份`。

### 自动备份流程

两个设计器在进入页面后启动一个 1 分钟定时器。

每次执行：

1. 生成当前 `pageConfig`
2. 计算 `currentHash`
3. 如果 `currentHash === lastAutoBackupHash`，直接跳过
4. 如果不同，则调用 `pageApi.historyBackup`
5. 备注固定为 `自动备份`
6. 成功后更新 `lastAutoBackupHash = currentHash`

注意：

- 自动备份不要求设计态已保存
- 自动备份失败时不阻断编辑
- 自动备份失败后不更新 `lastAutoBackupHash`，下一轮继续尝试

### 设计器历史按钮与弹窗

两个设计器顶部按钮顺序统一为：

- 变量
- 图层
- 设置
- 历史
- 预览
- 保存

其中 `历史` 位于 `预览` 左侧，满足需求描述。

点击 `历史` 后弹出对话框，首版使用表格分页即可，不引入额外筛选。

字段：

- `时间`
  - 直接显示 `createDate`
- `备注`
  - 直接显示 `remark`
- `操作`
  - `回滚`

交互规则：

- 弹窗打开时默认查询第一页
- 页码切换时重新请求数据
- 查询结果按时间倒序显示

### 回滚流程

点击某条历史后的 `回滚` 按钮时，前端流程固定为：

1. 重新计算 `currentHash`
2. 如果 `currentHash !== lastSavedHash`：
   - 不弹确认框
   - 直接提示“当前有未保存改动，请先保存后再回滚”
   - 流程结束
3. 如果没有未保存改动：
   - 弹确认框
   - 文案明确说明将先备份当前设计态，再回滚到所选历史版本
4. 用户确认后调用 `pageApi.historyRollback(id)`
5. 成功后提示成功
6. 调用页面刷新或重新拉取设计态数据

本期按用户要求采用整页刷新，保持行为最简单、最稳。

刷新后需重新初始化：

- `pageStageEntity`
- `chartList`
- `basicConfig` / `pageBasicConfig`
- `globalVariable`
- `lastSavedHash`
- `lastAutoBackupHash`

### 列表展示所有 history 的影响

用户已经确认弹窗展示所有 `history`，因此弹窗中会出现：

- 手动保存自动备份
- 自动备份
- 回滚前自动备份
- 发布自动备份
- 下线相关历史

本期不做类型分组，也不做标签化展示。用户通过 `备注` 自行识别来源。

这样做的原因是：

- 与后端现状兼容性最好
- 本期无需额外改表
- 先满足“可回看、可回滚、不丢数据”的核心诉求

## 共享前端模块建议

建议新增一个共享模块，专门服务两个设计器：

- `stableStringifyPageConfig(pageConfig)`
- `buildPageConfigHash(pageConfig)`
- `createDesignerHistoryAutoBackup(options)`

其中自动备份模块职责保持最小：

- 接收一个 `getPageConfig` 回调
- 接收一个 `backup` 回调
- 内部维护轮询启停
- 只负责“每分钟检查一次并在变化时触发备份”

这样可以避免在两个设计器里复制一整套定时检查代码。

## 错误处理

### 自动备份失败

- 不阻断当前编辑
- 仅做轻提示或控制台记录
- 不更新 `lastAutoBackupHash`
- 等待下一分钟继续重试

### 手动保存失败

- 提示保存失败
- 不新增历史记录
- 不更新 `lastSavedHash`
- 不更新 `lastAutoBackupHash`

### 历史列表加载失败

- 弹窗中展示加载失败状态
- 提供重试能力

### 回滚失败

- 提示回滚失败
- 页面不刷新
- 不修改前端本地摘要状态

## 测试设计

### 后端测试

至少覆盖以下场景：

- 调用历史备份接口后，新增一条 `history` 记录
- 相同内容重复调用历史备份接口时，仍然新增新记录
- 快照式回滚时，会先新增一条 `回滚前自动备份`
- 快照式回滚后，当前 `design.pageConfig` 被目标历史内容覆盖
- 快照式回滚后，目标历史记录自身保持 `history` 不变
- 历史列表查询按 `createDate desc` 返回

### 前端测试

至少覆盖以下场景：

- 两个设计器顶部都能展示 `历史` 按钮
- 打开历史弹窗后能按页加载历史数据
- 手动保存成功后会调用历史备份接口
- 自动备份定时器在内容未变化时不会重复备份
- 自动备份定时器在内容发生变化时会新增一条历史
- 当前存在未保存改动时，点击回滚会被拦截
- 回滚成功后页面刷新并重新加载设计态

## 实施边界

本需求是一个单独的历史备份能力，不与当前会话内 `undo/redo` 历史栈耦合。

两者职责不同：

- `undo/redo`：当前会话内、细粒度、本地可逆操作
- 历史自动备份：跨崩溃、跨刷新、后端持久化快照

本期实现中，二者只共享设计器页面入口，不共享数据结构和触发时机。

## 结论

本期采用“后端持久化历史快照 + 前端一分钟轮询差异检查 + 顶部历史弹窗 + 快照式回滚”的方案。

核心原则如下：

- 手动保存每次都留历史
- 自动备份只在内容变化时留历史
- 回滚前必须无未保存改动
- 回滚时先备份当前设计态，再覆盖设计态
- 目标历史记录保持不变
- 两个设计器共用同一套历史规则

这个方案能在不改表结构的前提下，补齐断电、崩溃、误关闭场景下的恢复能力，同时保证回滚可逆、行为稳定、实现边界清晰。
