# 功能规格说明：融合定时器与组件交互动作配置

**功能分支**: `[002-unify-timer-interactions]`

**创建时间**: 2026-06-28

**状态**: 草稿

**输入**: 用户描述："优化大屏设计器和页面设计器中定时器的动作配置逻辑。定时器本质也是一种触发事件，与图表组件的交互中的事件一样可以配置动作。需要融合定时器和组件的交互配置；不考虑旧配置兼容；但 ChartConfig 类型不允许修改、组件 install.ts 不允许修改，现有命名暂时不用修改。"

## 宪章对齐信息 *(必填)*

### 影响表面

- **架构表面**: 影响前端页面设计器与大屏设计器的定时器配置、组件交互动作配置、画布交互触发入口、定时器运行管理和页面基础配置中的定时器结构；不涉及后端模块、JSON 子类型注册、数据集工厂、路由或组件自动注册机制。
- **契约同步范围**: 需要同步 `PageTimer` 定时器配置契约、定时器配置弹窗、组件交互配置弹窗、动作配置公共编辑能力、画布实例动作执行入口、定时器运行触发逻辑和相关测试；明确不修改 `ChartConfig` 类型、不修改各组件 `install.ts` 导出的 `behaviors` 契约、不重命名现有 `Behavior`、`ChartAction`、`triggerChartAction` 等概念。
- **设计规范范围**: 适用 `docs/design/DESIGN.md`；受影响 UI 包括组件交互动作配置弹窗、定时器配置弹窗、动作列表、动作表单、目标组件选择入口和事件参数说明区域。样式必须继续使用 Element Plus CSS 变量，不新增硬编码颜色或覆盖 Element Plus 内部样式。
- **验证范围**: 前端必须运行 `cd dataRoomFront && npm run type-check`；涉及样式、模板结构或格式敏感改动时运行 `cd dataRoomFront && npm run lint`。本功能不涉及后端 Java 逻辑，默认不需要 Maven 测试或 `CatchBlockLoggingTest`。
- **安全与可追踪范围**: 不改变用户角色、鉴权、审计字段、软删除、RSA 密码处理或后端异常日志要求。前端动作执行失败必须保持可追踪反馈，不应静默吞掉定时器动作或组件交互动作的执行错误。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 定时器复用完整动作配置能力（优先级：P1）

作为大屏开发者，我希望配置定时器时能使用与组件交互相同的动作能力，以便通过 5 秒、10 秒等不同定时器分别驱动数据刷新、组件显隐、页面跳转、全局变量更新或代码块执行。

**优先级原因**: 这是本功能的核心价值；当前定时器动作配置被单独实现且主要限制在代码块，无法覆盖组件交互已有的动作类型。

**独立测试方式**: 在页面设计器或大屏设计器中创建一个 5 秒定时器，为其配置刷新数据、显隐切换和代码块动作，预览页面并观察动作按间隔触发。

**验收场景**:

1. **给定** 用户已进入页面设计器或大屏设计器并存在至少一个可选组件，**当** 用户新增一个 5 秒定时器并配置“数据刷新”动作，**则** 系统允许选择目标组件并在预览中按 5 秒间隔刷新目标组件数据。
2. **给定** 用户已创建一个定时器，**当** 用户为该定时器配置“显示”“隐藏”“显隐切换”“页面跳转”“更新全局变量”或“代码块”动作，**则** 系统提供与组件交互动作一致的配置体验，并按动作列表顺序执行。
3. **给定** 一个定时器配置了多个动作，**当** 定时器触发时，**则** 系统按照用户配置的顺序依次执行全部未禁用的动作。

---

### 用户故事 2 - 多个不同间隔定时器独立触发（优先级：P1）

作为大屏开发者，我希望同一页面中多个定时器可以拥有不同间隔和独立动作列表，以便分别实现高频数据刷新、低频状态轮播和周期性全局变量更新。

**优先级原因**: 定时器作为事件触发源时必须能区分不同实例，否则无法支撑真实页面中多个定时任务并存的场景。

**独立测试方式**: 创建 5 秒和 10 秒两个定时器，分别绑定不同动作，预览时验证两个定时器按各自间隔独立触发且互不覆盖配置。

**验收场景**:

1. **给定** 页面中存在一个 5 秒定时器和一个 10 秒定时器，**当** 页面进入预览或运行态，**则** 两个定时器分别按照自己的间隔触发 `tick` 事件。
2. **给定** 两个定时器都拥有 `tick` 事件动作，**当** 任一定时器触发，**则** 系统只执行该定时器实例下绑定的动作列表，不执行其他定时器的动作。
3. **给定** 用户禁用其中一个定时器，**当** 页面运行，**则** 被禁用的定时器不再触发动作，其他启用定时器继续正常工作。

---

### 用户故事 3 - 组件交互配置体验不退化（优先级：P2）

作为大屏开发者，我希望组件交互配置保持现有行为和入口不变，以便已有的点击、图例点击、行点击等组件事件仍能使用原来的动作配置能力。

**优先级原因**: 本功能要融合公共能力，但不能破坏现有组件交互配置体验；组件交互是设计器已有核心能力。

**独立测试方式**: 选择任意支持交互的组件，打开交互配置，配置点击事件动作，并在预览中触发组件点击验证动作执行。

**验收场景**:

1. **给定** 组件 `install.ts` 仍导出原有 `behaviors`，**当** 用户打开组件控制面板的交互页，**则** 系统继续展示该组件支持的交互事件。
2. **给定** 用户为组件点击事件配置动作，**当** 组件点击事件在预览中触发，**则** 系统继续执行该组件事件绑定的动作列表。
3. **给定** 本功能完成后，**当** 检查组件配置结构，**则** `ChartConfig` 类型和组件 `install.ts` 的事件定义契约没有被修改。

---

### 用户故事 4 - 定时器动作可使用事件参数（优先级：P3）

作为高级配置用户，我希望定时器触发动作时能获得定时器事件参数，以便在代码块或更新全局变量动作中区分触发来源和触发时间。

**优先级原因**: 事件参数使定时器真正成为标准触发事件，也让代码块和全局变量赋值能力与组件交互保持一致。

**独立测试方式**: 为定时器配置代码块动作，读取 `timerId`、`timerName`、`interval` 和 `timestamp` 参数并验证运行态值符合当前定时器。

**验收场景**:

1. **给定** 用户为定时器配置代码块动作，**当** 定时器触发，**则** 代码块可以读取当前定时器的 ID、名称、间隔和触发时间戳。
2. **给定** 用户为定时器配置更新全局变量动作，**当** 定时器事件参数被选为赋值来源，**则** 系统按该参数更新目标全局变量。

### 边界情况

- 页面没有任何组件时，定时器动作编辑器应允许配置不依赖目标组件的动作，并对需要目标组件的动作展示空选择状态。
- 用户删除某个目标组件后，已配置到该组件的定时器动作或组件交互动作用例应提示目标组件不存在，并跳过或阻止无效执行，不应导致页面崩溃。
- 定时器间隔小于系统允许的最小值时，系统应阻止保存并提示用户修正。
- 定时器存在但 `tick` 事件被禁用时，该定时器可以继续保留配置，但运行态不执行其动作列表。
- 多个定时器同时触发时，系统应按各自触发源实例执行动作，不应串用其他定时器或组件事件的参数。
- 动作列表为空时，定时器触发或组件事件触发不应报错。
- 动作执行失败时，系统应提供可追踪错误反馈，并继续保持设计器或预览页面可用。
- 页面配置重新加载、历史回退或定时器配置变化后，旧定时器实例应被正确停止，避免重复触发。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统 MUST 将定时器建模为页面级触发源，每个定时器实例 MUST 拥有独立的 `tick` 触发事件配置。
- **FR-002**: 系统 MUST 支持同一页面配置多个定时器实例，每个定时器实例 MUST 能独立设置名称、启用状态、执行间隔和 `tick` 事件动作列表。
- **FR-003**: 系统 MUST 在定时器动作配置中复用组件交互已有的动作类型能力，包括显示、隐藏、显隐切换、页面跳转、数据刷新、更新全局变量和代码块。
- **FR-004**: 系统 MUST 在定时器配置 UI 中提供与组件交互动作一致的动作列表管理能力，包括新增、删除、排序、动作命名和动作类型配置。
- **FR-005**: 系统 MUST 在定时器动作配置中复用目标组件选择能力，使需要目标组件的动作可以选择一个或多个目标组件。
- **FR-006**: 系统 MUST 在定时器动作配置中提供定时器 `tick` 事件参数说明，至少包含 `timerId`、`timerName`、`interval` 和 `timestamp`。
- **FR-007**: 系统 MUST 在定时器触发时向动作执行逻辑传入当前定时器实例的事件参数，使代码块和更新全局变量动作可以读取这些参数。
- **FR-008**: 系统 MUST 提供公共触发入口，使图表组件事件和定时器事件都能通过统一流程查找动作列表并执行动作。
- **FR-009**: 系统 MUST 保留现有组件交互配置契约，组件仍通过 `install.ts` 导出 `behaviors`，组件配置仍使用 `ChartConfig.behaviors` 保存交互动作。
- **FR-010**: 系统 MUST NOT 修改 `ChartConfig` 类型定义。
- **FR-011**: 系统 MUST NOT 修改各图表组件 `install.ts` 中的 `behaviors` 导出契约。
- **FR-012**: 系统 SHOULD 保留现有 `Behavior`、`ChartAction`、`triggerChartAction` 等命名，避免本次改动扩大为命名迁移。
- **FR-013**: 系统 MUST 在定时器运行管理中只负责定时器生命周期和事件触发，不应在定时器管理器中单独判断或执行某一种动作类型。
- **FR-014**: 系统 MUST 在定时器配置变化、页面配置重新加载或设计器卸载时停止旧定时器实例，避免重复触发或内存泄漏。
- **FR-015**: 系统 MUST 保持组件交互现有用户体验不退化，用户仍可从组件控制面板进入交互配置并执行原有组件事件动作。
- **FR-016**: 系统 MUST 在动作执行失败、目标组件不存在或事件配置无效时提供可理解反馈，并避免导致设计器或预览页面崩溃。

### 关键实体 *(若功能涉及数据则必须填写)*

- **定时器实例**: 页面级触发源，表示一个按固定间隔触发的定时任务，包含唯一 ID、名称、启用状态、执行间隔和 `tick` 事件动作配置。
- **定时器 Tick 事件**: 定时器实例上的固定触发事件，表示一次间隔到达；该事件包含可供动作使用的参数，如定时器 ID、定时器名称、执行间隔和触发时间戳。
- **组件交互事件**: 图表组件已有触发事件，继续由组件 `install.ts` 的 `behaviors` 定义，并继续保存在 `ChartConfig.behaviors` 中。
- **动作配置**: 用户为某个触发事件配置的动作列表，继续使用现有动作类型，包含动作名称、动作类型和对应动作参数。
- **公共触发入口**: 运行态统一处理触发源、事件名称、事件参数和动作执行的能力，支持组件触发源和定时器触发源。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**: 用户可在 3 分钟内创建一个 5 秒定时器，并完成至少一个目标组件数据刷新动作配置。
- **SC-002**: 用户可在同一页面创建 5 秒和 10 秒两个定时器，预览时两个定时器均按各自间隔独立触发，且动作互不串扰。
- **SC-003**: 定时器动作配置支持的动作类型覆盖率达到组件交互动作配置的 100%。
- **SC-004**: 本功能完成后，组件交互配置主流程仍可正常完成，至少覆盖一个组件点击事件动作配置和预览触发验证。
- **SC-005**: 定时器代码块动作可在运行态读取 `timerId`、`timerName`、`interval` 和 `timestamp` 四类事件参数。
- **SC-006**: 前端类型检查通过，且无因本功能引入的设计器预览页面崩溃问题。

## 假设前提

- 本功能不考虑旧页面配置兼容，旧的 `PageTimer.actions` 结构可以被新的定时器事件结构替代。
- 本功能只调整前端设计器和运行态交互逻辑，不引入后端接口、数据库结构或页面发布生命周期变更。
- 图表组件已有交互事件定义继续保持不变，不要求批量修改组件 `install.ts`。
- `ChartConfig` 类型作为现有组件配置契约保持不变，本功能通过定时器配置结构和公共触发入口完成融合。
- 现有动作执行能力足以覆盖定时器场景，本功能优先复用现有 `ChartAction` 和 `triggerChartAction` 逻辑。
- 定时器固定提供 `tick` 事件；定时器启动、停止、错误等额外事件不纳入本次范围。
- 用户主要在桌面端设计器中配置定时器和组件交互，移动端编辑体验不在本次范围内。
