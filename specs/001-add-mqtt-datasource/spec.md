# 功能规格说明：新增 MQTT 数据源

**功能分支**: `[001-add-mqtt-datasource]`

**创建时间**: 2026-06-20

**状态**: 草稿

**输入**: 用户描述："我想新增一个mqtt数据源"

## Clarifications

### Session 2026-06-20

- Q: MQTT 数据集执行时如何获得消息？ → A: 系统持续订阅并缓存每个 MQTT 数据集的最新有限消息，数据集执行时读取缓存结果。
- Q: MQTT 数据集默认缓存多少消息？ → A: 创建 MQTT 数据集时由用户配置缓存条数，默认缓存最近 5 条消息。
- Q: MQTT 消息体首版支持哪些格式？ → A: 首版仅支持 JSON 消息体，并从 JSON 字段映射为数据集字段。
- Q: MQTT 数据源首版支持哪些连接安全方式？ → A: 首版支持无认证、账号密码和 TLS 单向认证。

## 宪章对齐信息 *(必填)*

### 影响表面

- **架构表面**: 影响数据源管理、数据集执行能力、数据源类型反序列化注册、数据源连接校验、MQTT 订阅与最新消息缓存、前端数据源配置表单、数据集配置入口和相关权限校验。
- **契约同步范围**: 需要同步数据源类型定义、创建与编辑载荷、详情展示、连接测试结果、数据集引用 MQTT 数据源时的配置契约、持久化配置快照、接口文档和测试；涉及 HTTP 接口时响应包装继续使用 `{"code": 0, "msg": "success", "data": <T>}`。
- **设计规范范围**: 适用 `docs/design/DESIGN.md`；受影响 UI 包括数据源新增/编辑表单、连接测试反馈、数据集选择数据源时的类型展示和错误提示。
- **验证范围**: 后端需运行数据源与数据集相关 Maven 测试；如实现涉及 Java `catch` 块，必须运行 `mvn -q -pl dataRoomServer -Dtest=CatchBlockLoggingTest -DforkCount=0 test`；前端需运行 `npm run type-check`，涉及样式或格式变更时运行 `npm run lint`。
- **安全与可追踪范围**: 影响数据源凭据保存、TLS 单向认证材料保护、连接失败日志、角色权限、审计字段与软删除行为；不得削弱现有鉴权、审计和异常日志要求。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 管理员创建 MQTT 数据源（优先级：P1）

作为拥有数据源管理权限的用户，我需要在数据源管理页面选择 MQTT 类型并填写连接信息，以便把 MQTT 消息服务纳入 DataRoom 的统一数据源管理。

**优先级原因**: 没有可创建、可保存的 MQTT 数据源，后续数据集读取和大屏使用都无法成立。

**独立测试方式**: 通过创建一个 MQTT 数据源、保存后重新打开详情，即可验证配置录入、持久化、脱敏展示和基础管理能力。

**验收场景**:

1. **给定** 用户拥有数据源管理权限，**当** 用户选择 MQTT 类型并填写无认证、账号密码或 TLS 单向认证所需的必填连接信息后保存，**则** 系统成功创建数据源并可在列表中按名称和类型识别。
2. **给定** 已创建 MQTT 数据源，**当** 用户打开详情或编辑页，**则** 系统展示非敏感配置并对敏感凭据进行脱敏，不向普通页面明文暴露密码、令牌或证书密钥。

---

### 用户故事 2 - 连接测试与错误定位（优先级：P1）

作为配置 MQTT 数据源的用户，我需要在保存前或保存后测试连接，以便确认地址、认证信息、主题和订阅参数是否可用，并能根据错误提示完成修正。

**优先级原因**: MQTT 连接受网络、鉴权、主题权限和服务端策略影响，缺少测试能力会显著增加排障成本。

**独立测试方式**: 对一个可访问和一个不可访问的 MQTT 服务分别执行连接测试，验证成功反馈和失败提示是否准确、可理解、可追踪。

**验收场景**:

1. **给定** MQTT 服务可访问且配置有效，**当** 用户发起连接测试，**则** 系统在 10 秒内返回成功结果，并说明连接与订阅校验已通过。
2. **给定** 地址、认证或主题权限无效，**当** 用户发起连接测试，**则** 系统返回可操作的失败原因，并记录可供管理员定位的失败信息。

---

### 用户故事 3 - 数据集使用 MQTT 消息（优先级：P2）

作为大屏开发者，我需要在数据集中选择已配置的 MQTT 数据源并指定 JSON 消息读取规则，以便可视化组件可以展示来自 MQTT 主题的最新消息数据。

**优先级原因**: 数据源只有被数据集消费后，才能转化为大屏设计器中的业务价值。

**独立测试方式**: 使用一个已有 MQTT 数据源创建数据集，发送 JSON 测试消息后运行数据集，验证返回数据符合用户配置的 JSON 字段与缓存消息范围。

**验收场景**:

1. **给定** 已有有效 MQTT 数据源，**当** 用户创建数据集并选择主题、缓存消息条数和 JSON 字段解析规则，**则** 系统保存数据集并按配置维护该数据集的最新有限消息，默认缓存最近 5 条消息。
2. **给定** MQTT 主题收到符合规则的新消息，**当** 用户运行该数据集，**则** 系统从最新有限消息缓存中返回可被图表组件绑定的结构化数据。

---

### 用户故事 4 - 在大屏中稳定展示 MQTT 数据（优先级：P3）

作为大屏浏览者，我需要图表在预览和发布页面中稳定展示 MQTT 数据，以便实时或近实时业务状态可以被持续观察。

**优先级原因**: 发布后的展示稳定性决定 MQTT 数据源是否能用于真实监控场景。

**独立测试方式**: 将 MQTT 数据集绑定到一个图表组件，发布页面后持续发送消息，验证页面显示、刷新和异常状态符合预期。

**验收场景**:

1. **给定** 图表已绑定 MQTT 数据集，**当** 页面预览或发布后接收到新消息，**则** 图表在配置的刷新周期内展示最新可用数据。
2. **给定** MQTT 服务短暂不可用，**当** 页面刷新数据，**则** 页面不崩溃，并向有权限的用户提供可理解的异常状态。

### 边界情况

- MQTT 服务地址不可达、连接超时或服务端拒绝连接时，系统应返回失败状态并保留可追踪日志。
- 用户缺少数据源管理权限时，不得创建、编辑、删除或查看敏感 MQTT 配置。
- 主题为空、主题格式不符合要求或当前凭据无订阅权限时，连接测试和数据集预览应阻止继续并给出明确提示。
- 消息体为空、不是 JSON、JSON 结构不符合字段解析规则或字段解析失败时，数据集执行应返回可理解错误或空结果，不应导致页面崩溃。
- 数据源被删除或禁用后，引用它的数据集和页面应显示明确的不可用状态。
- 高频消息到达时，系统应按用户配置的数据范围返回可控结果，避免单次预览或展示加载无限消息。
- 数据集首次创建后尚未收到任何 MQTT 消息时，预览和图表展示应返回明确的空数据状态。
- 凭据更新后，旧凭据不应继续被新连接使用，详情页仍需保持脱敏展示。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统 MUST 支持用户在数据源管理中选择 MQTT 作为数据源类型。
- **FR-002**: 系统 MUST 要求 MQTT 数据源填写连接地址、端口、连接安全选项、认证信息、默认主题或主题规则、连接超时时间和说明名称；首版连接安全选项 MUST 覆盖无认证、账号密码和 TLS 单向认证。
- **FR-003**: 系统 MUST 在创建和编辑 MQTT 数据源时校验必填项、取值范围和主题规则格式，并在保存失败时给出明确原因。
- **FR-004**: 系统 MUST 对 MQTT 数据源中的敏感凭据和 TLS 单向认证材料进行受保护保存和脱敏展示，避免在列表、详情、错误消息和普通日志中泄露明文凭据或认证材料。
- **FR-005**: 用户 MUST 能够对 MQTT 数据源执行连接测试，并获得成功、超时、认证失败、主题无权限或配置错误等可区分结果。
- **FR-006**: 系统 MUST 允许有权限用户查看、编辑、删除和搜索 MQTT 数据源，并保持现有数据源管理的审计与软删除行为。
- **FR-007**: 系统 MUST 允许数据集引用已创建的 MQTT 数据源，并配置主题、最新有限消息缓存条数、JSON 字段解析规则和空数据处理方式；未调整时默认缓存最近 5 条消息。
- **FR-008**: 系统 MUST 持续订阅 MQTT 数据集配置的主题并维护最新有限消息缓存，数据集预览和执行时从缓存中返回可供可视化组件使用的结构化数据。
- **FR-009**: 系统 MUST 在 MQTT 数据源不可用、数据集配置无效、消息体不是 JSON 或 JSON 解析失败时返回可理解的错误结果，并记录完整可追踪的失败信息。
- **FR-010**: 系统 MUST 保持现有角色权限边界，只有具备相应权限的用户可以管理 MQTT 数据源或查看其非敏感配置。
- **FR-011**: 系统 MUST 在数据源列表、详情和数据集配置入口中清晰标识 MQTT 类型，便于用户区分不同数据源。
- **FR-012**: 系统 MUST 为 MQTT 数据源相关接口和用户操作提供与现有数据源一致的成功、失败和校验反馈。

### 关键实体 *(若功能涉及数据则必须填写)*

- **MQTT 数据源**: 表示一个可被 DataRoom 管理和复用的 MQTT 连接配置，包含名称、连接端点、连接安全选项、认证信息、TLS 单向认证材料、默认主题规则、超时设置、状态、创建人与更新时间等信息。
- **MQTT 数据集配置**: 表示某个数据集如何从 MQTT 数据源读取 JSON 消息，包含引用的数据源、主题、最新有限消息缓存条数、JSON 字段解析规则、空数据策略和预览状态；缓存条数默认值为 5。
- **最新消息缓存**: 表示系统为每个 MQTT 数据集维护的有限消息集合，用于数据集预览、执行和页面刷新读取，默认保留最近 5 条消息。
- **连接测试结果**: 表示一次 MQTT 数据源连通性验证的结果，包含测试状态、用户可读提示、失败分类和发生时间。
- **消息样本**: 表示用于预览和字段映射的 MQTT JSON 消息内容，包含原始消息、解析后的字段和值。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**: 具备数据源管理权限的用户可在 3 分钟内完成一个 MQTT 数据源的创建、连接测试和保存。
- **SC-002**: 对可访问的 MQTT 服务执行连接测试时，95% 的测试在 10 秒内返回明确结果。
- **SC-003**: 使用有效 MQTT 数据源创建数据集并收到匹配消息后，用户可在 5 分钟内完成一次消息预览并获得来自最新有限消息缓存的结构化结果。
- **SC-004**: MQTT 服务不可用或配置错误时，100% 的失败结果都包含用户可理解的错误提示，且不会暴露明文敏感凭据。
- **SC-005**: 已绑定 MQTT 数据集的页面在数据源短暂不可用时不发生页面崩溃，恢复后可继续展示最新可用数据。
- **SC-006**: 首轮内部验收中，至少 90% 的测试用户能够独立完成 MQTT 数据源创建与数据集预览主流程。

## 假设前提

- MQTT 数据源属于现有数据源管理体系的一种新类型，复用当前用户、角色、审计、软删除和数据源列表管理体验。
- 首个版本聚焦读取 MQTT 主题消息并供数据集和图表展示使用，不包含向 MQTT 主题发布消息的能力。
- 首个版本仅支持 JSON 消息体；纯文本和二进制消息体不在首版范围内。
- 首个版本支持无认证、账号密码和 TLS 单向认证连接场景，不包含双向证书认证。
- 数据集默认读取系统持续订阅后缓存的最新有限消息，默认缓存最近 5 条消息，不承担无限历史消息存储能力。
- 大屏展示继续使用现有数据集刷新机制，用户可通过现有组件配置控制刷新节奏。
- MQTT 服务、网络连通性和主题访问权限由用户或部署环境提供，DataRoom 负责配置管理、连接验证、数据读取和错误反馈。
