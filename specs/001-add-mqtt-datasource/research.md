# Phase 0 研究：新增 MQTT 数据源

## 决策 1：MQTT 客户端能力边界

**Decision**: 实现阶段选择维护中、兼容 JDK 17/Spring Boot 3、支持 MQTT 3.1.1/5、TLS 单向认证、自动重连、异步消息回调和连接状态监听的 Java MQTT 客户端。

**Rationale**: 本功能需要持续订阅主题并维护最新消息缓存，客户端必须能稳定处理重连、TLS 和异步消息；该能力不适合手写协议层。

**Alternatives considered**:

- 手写 MQTT 协议处理：风险高，协议和 TLS 细节复杂，不符合项目务实原则。
- 仅支持阻塞式临时订阅：实现简单，但与已澄清的“持续订阅并缓存”需求不一致。

## 决策 2：数据集执行模型

**Decision**: MQTT 数据集由运行态持续订阅主题并维护每数据集最新有限消息缓存；`run` 和 `run4Chart` 读取缓存返回结构化结果。

**Rationale**: 该模型与现有图表轮询/刷新数据流兼容，避免每次图表刷新都新建 MQTT 连接并等待消息，能让连接状态、缓存边界和错误反馈可测试。

**Alternatives considered**:

- 每次执行临时连接订阅：延迟和失败率更高，图表刷新体验不稳定。
- 浏览器端实时订阅：会暴露更多连接信息和凭据，且需要新的推送契约。

## 决策 3：缓存策略

**Decision**: 每个 MQTT 数据集配置可设置最新消息缓存条数，默认 5 条；缓存只保存运行态最新消息，不承担历史消息存储。

**Rationale**: 默认 5 条符合用户澄清，能够满足预览和常见图表展示，同时控制内存占用。历史消息属于另一类存储需求，应另行设计。

**Alternatives considered**:

- 固定 1 条：过于有限，不利于表格或趋势类展示。
- 固定 100/1000 条：默认内存成本更高，且与用户澄清不一致。

## 决策 4：消息体格式

**Decision**: 首版仅支持 JSON 消息体，通过 JSON 字段解析规则映射为数据集字段。

**Rationale**: DataRoom 图表绑定依赖结构化字段；JSON 是 MQTT 业务消息中常见且可直接映射的数据格式。限定首版范围能降低解析歧义并提升验收可测性。

**Alternatives considered**:

- 支持纯文本单字段：会引入字段语义弱化和更多 UI 分支。
- 支持二进制：需要编码、预览和安全限制，超出首版用户价值。

## 决策 5：连接安全范围

**Decision**: 首版支持无认证、账号密码和 TLS 单向认证；不支持双向证书认证。

**Rationale**: 该范围覆盖常见内网和基础安全部署，又避免首版引入客户端证书生命周期、密钥口令和双向认证排障复杂度。

**Alternatives considered**:

- 只支持无认证/账号密码：对启用 TLS 的部署覆盖不足。
- 直接支持双向证书认证：安全能力更完整，但 UI、凭据保存、测试环境和排障复杂度显著增加。

## 决策 6：敏感信息处理

**Decision**: MQTT 密码、TLS 单向认证材料等敏感信息必须受保护保存，详情、列表、错误消息和普通日志只返回脱敏值或状态，不返回明文。

**Rationale**: 符合 DataRoom 宪章的安全默认要求，也与现有数据源凭据处理模式一致。

**Alternatives considered**:

- 明文回显便于编辑：安全风险不可接受。
- 不允许编辑敏感字段：可行但体验较差，首版采用“留空表示不更新、填写表示替换”的常见模式更合适。

## 决策 7：前端 UI 实现约束

**Decision**: MQTT 数据源和数据集配置 UI 使用 Element Plus 默认表单组件，通过 props 表达状态和密度；业务样式只控制外层布局、间距和滚动区域。

**Rationale**: 项目设计规范明确禁止覆盖 Element Plus 内部样式、硬编码颜色和新增私有颜色 token。

**Alternatives considered**:

- 为 MQTT 表单单独定制视觉：不符合 `docs/design/DESIGN.md`，会造成设计系统漂移。
